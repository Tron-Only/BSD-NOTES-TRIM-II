# SDLC PHASE 1: System Planning
This is where requirements and problems are defined in a System Requirements document so that it can be sent over to the next phase for analysis.

Requirements could be things like cost, estimate of resources or personnel
Problems could be caused by two factors, internal or external
### Internal Factors
1. Inadequate functionality
2. Poor performance
3. Excessive workload
### External Factors
1. Changes in the market
2. Change in customer needs/demands
3. Deviating from established practices 

After identifying problems, a System Request is output

> [!note] System Requirement
> A request that is made after having identified and defined a problem


Requests can be major or minor.
Major requests involve creating an entirely new system to either replace the old system or create a system that a business needs
Minor requests involve adjusting an already predefined system.

## Activities Involved in System Planning
### Identifying the System Development Project
There are two ways to identify a problem:
- Top-down planning - This is where management ask the IT support by sending a system service request to either improve a system, fix errors or bugs, or make an entirely new system.
- Bottom-up planning - This is where users ask for IT support by sending a system service request.

> [!note] System Service Request
> A formal way for asking for IT support

#### Reasons why users would ask for a system service request
1. For IT support for new system features
2. To get more information out of the system
3. To improve services

### Planning the System Development Project
After getting permission from management the system analyst interacts with the stakeholders (system owner, project managers, system users) to gather information like problems, scope and constraints, benefits and development time and costs.
The analyst will then prepare a report for the management.

System analysts use tools like Microsoft Project to plan and manage their tasks, this is represented in a Gantt chart.


### Understanding the problem/opportunity
After getting a system service request, a team will attempt to understand the project.

Stakeholders, that is, system owners, project managers and system users, have different levels of understanding, therefore, the team will ask them different questions to gain understanding to leads to a possible new system. They analyzed the problem based on cause-and-effect analysis.

### Defining the project scope and constraints
Project Scope is defining the specific boundaries the system should have. 
Project Scope Definition is concerned by the system owners' view of the system and the problems/opportunities that come out with the interest of the system

Project Constraints conditions a system must achieve in order to achieve its goal.
Project Constraint Definition is derived by system hardware, software, time or organization policy
